C++ BigInteger

I have spend an embarrassing amount of time on this project over the last year. At this point I'm currently working on the 5th version lol.

The point of a library like BigInt is store store and manipulate arbitrary sized integers. I originally started working on this when working on, guess what, another project. yay! That other project, I can't share publicly but I'm working on something cool to show this off. The reason this exists is as follows, the language I use most often C++, is limited by it's number storage. Integers aka whole numbers are commonly stored in this fashion And because of this "hard" limit, you can only go so big. In my case, its about 19 digits or something. For one of my problems I had to add a couple hundred, 50 digit numbers, so thats where this all started. This library allows for an "infinitely" large number, you just need to have enough storage in memory.

Other languages have this functionality built in. Python is a good example, you can go as big as your computer will let you, straight out of the box. Other languages, like Java, you just have to import their package in your code. Its like getting a book from a library. Once you have the book, you have the knowledge inside. Java's BigInteger library is open and readable by the public, and version 3 was based off of it, but its way advanced for me to understand. Some true magic going on over there.

All prior 4 versions where based around structures that hold a list/array/string of digits. If you had the number 10, it would be represented as { 0, 1 }, it's "backward" for some simplicity reasons so I don't get even balder. I then program each operation. This includes arithmetic (+, -, *, / %), boolean (<, >, ==, !=), and some other quality of life ones.

This 5th version, and I feel stupid for not thinking of it sooner, is chunk based. Similarly as to how Minecraft loads its world in 16 x 16 chunks, I store my numbers in 9 digit chunks. (9 digits for the same balding reason) This is great for this reason. Lets say you don't use chunks and add 12 and 13. You have to add the 2 and 3, then the 1's. Which is ok, except when your computer is calculating the factorial of 1 million (which the new version can do with ease. slay) This number has around 5.5 million digits. Multiplication as you learned in like 2nd grade, is just some additions shifts for place. This method's operation count is ~n^2, n being the length of the number. Meaning on the last calculation alone, with this digit by digit, no chunk method, the computer would have to do 30,250,000,000,000 operations (30 trillion ish). Obv this isn't fast. But if we represent our number as chunks, we get to do 1/9 the operations when adding, and even less when muliplying.

Better yet, since we use chunks, if both numbers are less than 9 digits, we can add them normally, so we have good performance with small numbers too.

I profusely apoligize for this, that was a lot and I applaud you for reading all the way here. I really like stuff like this so I could go on and on.
